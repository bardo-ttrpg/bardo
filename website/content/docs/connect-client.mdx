Connect the client to the local bridge after the workspace is prepared.

## Connection flow

1. Run `bardo login` from the workspace root.
2. Approve the browser session.
3. Run `bardo init` if the workspace has not been prepared yet.
4. Run `bardo connect --client codex` or the matching client flag.
5. Open the client, inspect readiness/current state, then use the local runtime tools.

## Exact commands

```bash
cd ~/projects/my-campaign
bardo login
bardo init
bardo connect --client codex
```

If the rulebook is inside the workspace but not at the root:

```bash
bardo init --rulebook ./rules/core-rulebook.md
```

`bardo init` writes its prep layer into `.bardo/`, including:

- `.bardo/rules/rulebook.md`
- `.bardo/rules/normalized/index.json`
- `.bardo/manifests/source-index.json`
- `.bardo/entities/campaign-entities.json`
- `.bardo/state/current-state.json`
- `.bardo/simulation/tracking-profile.json`
- `.bardo/manifests/readiness.json`

## What you approve

The browser approval confirms account access for the bridge session. The hosted side enforces approvals, billing, entitlements, status checks, and token refresh. It does not own campaign truth.

## What stays local

Workspace discovery, rules bootstrap, campaign bootstrap, and runtime tool execution stay on your machine. The generated `.bardo/` artifacts are the local prep layer that clients should read before continuing play.

When the player explicitly corrects a fact, use `user_correction` so the correction becomes the highest-precedence local truth instead of staying buried in chat history.

## Session hygiene

Treat `429` responses from hosted routes as backpressure on approvals or status checks. The fix is to back off and retry, not to spam the bridge. Normal play and canon ownership remain local even when the hosted layer is temporarily unavailable.

Agents should behave like conservative TTRPG GMs: read the prep artifacts first, surface uncertainty, prefer safe partial answers, and avoid inventing canon when readiness or evidence is weak.
