Bardo is a local-first campaign runtime. The workspace is the source of truth, `.bardo/` is the only Bardo-managed root, and the hosted side stays focused on auth, approvals, billing, entitlements, and protected account logic.

## What Bardo covers

- Install the local bridge on the machine that holds your campaign files.
- Run `bardo login`, `bardo init`, and `bardo connect` from the workspace root.
- Generate a strict `.bardo/` prep layer with rules bootstrap and campaign bootstrap outputs.
- Use local runtime tools such as `scene_turn`, `player_action`, `user_correction`, `world_sync`, and `simulation_tick`.
- Keep the hosted boundary narrow and predictable.

## Recommended path

1. Install the CLI bridge locally.
2. Put `rulebook.md` in the workspace root if you want the default rules bootstrap path.
3. Run `bardo login`.
4. Run `bardo init`.
5. Read `.bardo/manifests/readiness.json` and `.bardo/state/current-state.json`.
6. Run `bardo connect --client <your-client>`.
7. Use the runtime tools against the local prep artifacts.
8. If the player explicitly corrects canon, use `user_correction` instead of hand-editing generated state.

`rulebook.md` on its own is enough to make Bardo safe and conservative, but not enough to imply a richly prepared world. Expect `ready-with-gaps` or limited continuity until campaign notes outside `.bardo/` provide grounded state.

## What stays outside docs

The website does not try to mirror every internal artifact. It documents the stable boundary: local workspace ownership, the generated `.bardo/` outputs, the public CLI entrypoints, and the hosted routes that remain server-backed.
