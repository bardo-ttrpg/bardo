Bardo is the MCP for playing any tabletop role-playing game. It works with many modern AI clients, keeps your campaign files local, and grounds the model in your real workspace so it stays far more accurate.

Your workspace is the source of truth, `.bardo/` is the only Bardo-managed folder, and the hosted side only handles account, approval, and billing tasks.

## What Bardo covers

- Install the local bridge on the machine that holds your campaign.
- Prepare the workspace with `bardo login`, `bardo init`, and `bardo connect`.
- Build a local `.bardo/` prep layer that keeps play grounded in your files.
- Keep account, approval, and billing logic outside your campaign files.
- Give the connected client a local MCP endpoint plus workspace skill guidance without turning Bardo into a hosted gameplay service.

## Recommended path

1. Install the CLI bridge locally.
2. Put `rulebook.md` in the workspace root if you want the default setup path.
3. Run `bardo login`.
4. Run `bardo init`.
5. Check readiness in `.bardo/manifests/readiness.json`.
6. Run `bardo connect --client <your-client>`.
7. Open your AI client and start from the workspace you just prepared.

Expected result: you should have a `.bardo/` folder, a readiness file, and a connected client that can work from the local campaign context instead of guessing.

The browser approval and hosted website exist so Bardo can verify account access, subscription state, and bridge approvals. Your campaign truth and local runtime state still stay on your machine.

`rulebook.md` on its own is enough to make Bardo safe and conservative, but not enough to create a rich world automatically. If your campaign notes are still thin, expect `ready-with-gaps` and simpler play until you add more grounded source material.

## What stays outside docs

This website only covers the stable user path: install, prepare, connect, and use Bardo safely. It does not try to teach every internal runtime detail.
