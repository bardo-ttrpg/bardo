Bardo keeps the play surface small and local. It works from the prepared `.bardo/` layer instead of sending your campaign to a large hosted gameplay API.

## Tool shape

Bardo separates setup actions from play actions. The important user-facing rule is simple: prepare the workspace first, then let the connected client use the local Bardo runtime from that prep layer.

## Local prep first

The setup flow prepares:

- `.bardo/rules/normalized/index.json`
- `.bardo/manifests/source-index.json`
- `.bardo/entities/campaign-entities.json`
- `.bardo/state/current-state.json`
- `.bardo/simulation/tracking-profile.json`
- `.bardo/manifests/readiness.json`
- `.bardo/events/state-changes.ndjson`

That prep layer is what keeps Bardo grounded. It should not be replaced with ad-hoc chat memory.

If `.bardo/` is missing, Bardo should stop and treat the workspace as uninitialized until `bardo init` is run again.

## Canon and commits

The important rule is that narration alone is not canon. Durable truth comes from grounded source material, validated local state-changing events, or an explicit user correction. When a correction is needed, Bardo records it through the `user_correction` path before the next turn.

## Guardrails

The bridge does not turn Bardo into a hosted gameplay service. The hosted side is there for approvals, auth, and billing only.

If Bardo cannot continue safely, it should stop, explain what is missing, and wait for better source material or an explicit correction.
