Bardo exposes a small runtime surface built on top of the local prep artifacts instead of a large remote gameplay API.

## Tool shape

The preferred public runtime tools are:

- `init`
- `scene_turn`
- `player_action`
- `user_correction`
- `world_sync`
- `simulation_tick`

The tools read from `.bardo/` artifacts, apply canon precedence conservatively, and only commit validated state-changing events or explicit user corrections.

## Local prep first

`init` is the entrypoint for rules bootstrap plus campaign bootstrap. It prepares:

- `.bardo/rules/normalized/index.json`
- `.bardo/manifests/source-index.json`
- `.bardo/entities/campaign-entities.json`
- `.bardo/state/current-state.json`
- `.bardo/simulation/tracking-profile.json`
- `.bardo/manifests/readiness.json`
- `.bardo/events/state-changes.ndjson`

Runtime tools should operate from those artifacts instead of improvising their own shadow state.

## Canon and commits

- `scene_turn` can narrate conservatively from the current state.
- Narration is not canon by itself.
- `user_correction` records explicit player corrections at the highest precedence.
- `player_action`, `world_sync`, and `simulation_tick` commit validated state-changing events to `.bardo/events/state-changes.ndjson`.
- The committed event stream and `state/current-state.json` form the local runtime truth.
- The default precedence order is: explicit user correction, preserved source rules text, current campaign source files, approved committed state, recent validated play result, inference, narration flavor.

## Guardrails

The bridge intentionally does not proxy a remote gameplay surface. Hosted routes still support approvals, introspection, billing, and runtime status, but campaign execution remains local.

Bardo should stop or ask when canon conflict, rule ambiguity, or missing state would change the outcome materially. It may continue conservatively only when uncertainty stays marked and does not corrupt canon.
