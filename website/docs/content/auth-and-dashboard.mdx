# Auth & Dashboard

The Bardo website uses [Clerk](https://clerk.com) for authentication. Clerk handles sign-up, sign-in, session management, and OAuth providers — no passwords stored on Bardo's servers.

## Authentication routes

| Route | Description |
|---|---|
| `/sign-in` | Sign in with email or OAuth provider |
| `/sign-up` | Create a new Bardo account |
| `/dashboard` | Protected — requires authentication |
| `/api/webhooks/clerk` | Internal — Clerk → Convex sync webhook |

## Protection model

Unauthenticated requests to `/dashboard` are intercepted by Clerk middleware and redirected to `/sign-in` with a return URL. After signing in, users land back at `/dashboard` automatically.

<Callout type="info">
  All other routes (`/`, `/mpc-docs`, `/pricing`) are publicly accessible without an account.
</Callout>

## Setting up auth (self-hosted)

If you're running your own instance of the Bardo website, follow these steps to configure Clerk:

<Steps>

### Create a Clerk application

Go to [clerk.com](https://clerk.com) and create a new application. Choose your sign-in methods (email, GitHub, Google, etc.).

### Copy your API keys

From the Clerk dashboard, copy the following values:

```bash filename=".env.local"
NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY=pk_live_...
CLERK_SECRET_KEY=sk_live_...
CLERK_JWT_ISSUER_DOMAIN=https://your-domain.clerk.accounts.dev
CLERK_WEBHOOK_SECRET=whsec_...
```

### Configure the webhook

In the Clerk dashboard under **Webhooks**, create a new endpoint pointing to your deployment:

```
https://your-domain.com/api/webhooks/clerk
```

Subscribe to these events:
- `user.created`
- `user.updated`
- `user.deleted`

### Set the sign-in/sign-up URLs

In Clerk's **Paths** settings, set:

| Setting | Value |
|---|---|
| Sign-in URL | `/sign-in` |
| Sign-up URL | `/sign-up` |
| After sign-in URL | `/dashboard` |
| After sign-up URL | `/dashboard` |

### Deploy and verify

Start your dev server and visit `/sign-up`. After creating an account, you should be redirected to `/dashboard` and see your user avatar in the navbar.

</Steps>

## Session UI

The navigation bar adapts based on auth state:

- **Signed out** — shows **Log in** and **Sign up** links
- **Signed in** — shows the Clerk `UserButton` avatar with account controls and sign-out

<Callout type="warning">
  The `IS_CLERK_CONFIGURED` flag in the codebase checks that all three required env vars (`publishableKey`, `secretKey`, `issuerDomain`) are present. If any are missing, the site falls back to static sign-in links rather than rendering broken Clerk components.
</Callout>

## User data in Convex

When a user signs up, Clerk sends a `user.created` webhook to `/api/webhooks/clerk`. The handler writes a new record to the Convex `users` table with:

```ts
{
  clerkId: "user_...",
  email: "user@example.com",
  plan: "free",
  creditsTotal: 100,
  creditsUsed: 0,
  periodStart: Date.now(),
}
```

This record powers the dashboard's credit display and gates access to paid MCP features.
